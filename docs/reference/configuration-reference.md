# Configuration Reference

This starter kit intentionally keeps most project-level customization in a small
set of files.

If you are adapting the template for your own app, start here.

## `lib/config/app_config.dart`

This is the main product-level configuration file.

### `appName`

The display name used across the app shell and starter UI.

Change this first.

### `bundleId`

Currently a placeholder:

```dart
com.example.starterkit
```

Update this to match your real app identifiers and your Firebase / App Store /
Play Store configuration.

### RevenueCat keys

- `revenueCatAppleApiKey`
- `revenueCatGoogleApiKey`

These should be RevenueCat public SDK keys for the matching platforms.

### Legal URLs

- `privacyPolicyUrl`
- `termsOfServiceUrl`

These drive the links shown in the settings screen.

### Feature flags

- `enablePaywall`
- `enableNotifications`

These are useful when you are incrementally wiring the project:

- leave them `false` if the integration is not ready yet
- switch them to `true` once the provider setup is complete

### Navigation

- `bottomNavTabCount`

This exists as a small starter constant. If you expand the home shell into a
full tabbed app, keep this in sync with your navigation model.

## `lib/config/environment.dart`

This file defines the runtime environment:

- `dev`
- `staging`
- `prod`

The environment is selected through `--dart-define=ENV=<value>`.

Examples:

```bash
flutter run --dart-define=ENV=dev
flutter build ios --dart-define=ENV=prod
```

The current implementation is intentionally minimal. If your team needs:

- different API endpoints
- different Firebase projects
- different feature flags by environment

you will probably expand this file and pair it with environment-specific
configuration classes.

## `lib/config/theme.dart`

This is the main visual customization point for the starter kit.

The current implementation controls:

- Material 3 usage
- light theme
- dark theme
- seed color
- font family

Typical first changes:

- replace the seed color
- replace the font family
- tune surfaces, outlines, and typography for your brand

## `lib/main.dart`

This is the startup orchestration point.

Current order:

1. initialize Flutter bindings
2. initialize environment
3. initialize Firebase
4. initialize RevenueCat if enabled
5. initialize FCM if enabled
6. run the app inside `ProviderScope`

If you later add:

- analytics
- crash reporting
- remote config
- Sentry
- feature flag providers

this is the file where those boot-time integrations will usually start.

## `lib/shared/services/firebase_service.dart`

This file currently wraps Firebase startup:

```dart
await Firebase.initializeApp();
```

For most real apps, replace this with `DefaultFirebaseOptions.currentPlatform`
generated by FlutterFire.

## `pubspec.yaml`

This file defines:

- the package name
- app version
- Dart SDK constraint
- Flutter/Firebase/Riverpod/RevenueCat dependencies

Important note:

The package version is currently aligned with the documented open-source release
at `0.1.0`.

## Safe Customization Order

If you are brand-new to the project, make changes in this order:

1. `app_config.dart`
2. `environment.dart`
3. `theme.dart`
4. `firebase_service.dart`
5. route guards and product-specific feature logic

That order gives you the fastest path to a working app without getting lost in
product-specific implementation too early.
